# Elasticsearch Backend Validation — CI/CD

> This file documents that the 8 Sigma rules in this directory are
> converted into real Lucene queries (via `pysigma-backend-elasticsearch`)
> and executed against a live Elasticsearch instance, automatically,
> on every push, via GitHub Actions — not just matched with a
> hand-written Python evaluator.

---

## Why this step matters

The offline test runner (`tests/test_rules.py`) evaluates rule logic
directly against Python dicts — useful for fast iteration, but it is
a custom evaluator, not a production detection engine. This step closes
that gap: the same Sigma YAML rules are converted by `pySigma` into
Lucene query strings and executed by a real Elasticsearch instance,
proving the rules are engine-compatible, not just logically consistent
in a bespoke script.

## Why CI instead of local Docker

An earlier iteration of this validation ran locally via Docker Compose
on WSL2. Given the 6GB RAM constraint of the development machine (2GB
allocated to WSL2), running Elasticsearch's JVM alongside Docker
overhead proved unstable and occasionally crashed the WSL2 instance.

Moving validation to GitHub Actions removes the hardware constraint
entirely: GitHub's standard runners provide 7GB RAM, free of charge,
for the duration of the job only — no persistent infrastructure to
maintain, no local resource contention. The same Sigma → pySigma →
Lucene → Elasticsearch pipeline runs identically, just on
infrastructure sized for the task instead of a personal laptop.

## How it works

Workflow: [`.github/workflows/sigma-elasticsearch-validation.yml`](../.github/workflows/sigma-elasticsearch-validation.yml)

Triggers automatically on any push touching `07_detection_content/rules/**`
or `07_detection_content/ci/**`, or manually via "Run workflow" in the
Actions tab.

Steps:
1. Spin up Elasticsearch 8.15.0 as a GitHub Actions **service container**
   (zero local installation, isolated per job run).
2. Install Python dependencies (`pysigma`, `pysigma-backend-elasticsearch`,
   `elasticsearch` client pinned to the matching 8.x major version).
3. Convert all 8 Sigma rules to Lucene queries
   (`07_detection_content/ci/convert_to_lucene.py`).
4. Index the Mordor dataset (`empire_launcher_vbs`, APT29 emulation
   Day 1) into the live Elasticsearch instance and execute each
   converted query against it (`07_detection_content/ci/run_against_elasticsearch.py`).
5. Upload `lucene_queries.json` and `elasticsearch_validation_report.md`
   as a downloadable workflow artifact.

## Verifiable evidence

Unlike a written claim, anyone can independently verify this validation:

- Go to the repository's **Actions** tab → select the latest
  `Sigma Rules — Elasticsearch Validation` run → download the
  `elasticsearch-validation-report` artifact.
- `lucene_queries.json` contains the real Lucene query generated by
  `pySigma` for each of the 8 rules.
- `elasticsearch_validation_report.md` contains the actual match count
  returned by a live Elasticsearch instance for each rule against the
  Mordor dataset.

This proves rule-to-query conversion correctness and successful
real-engine evaluation, on infrastructure anyone can inspect — no
local environment or trust in a static claim required.

## A note on a debugging detour

The first CI run failed with `Elasticsearch non raggiungibile`. An
AI coding assistant (GitHub Copilot) diagnosed it as a startup-timing
issue and suggested longer health-check timeouts. That fix didn't
solve it: a `curl` readiness check confirmed Elasticsearch was already
reachable within 1 second, while the Python script still failed.

The actual cause was a version mismatch: `requirements.txt` did not
pin the `elasticsearch` Python client, so pip installed the latest
major version (9.x) against an Elasticsearch 8.15.0 server. The fix
was pinning the client to `elasticsearch>=8.15.0,<9.0.0` to match the
server's major version, plus switching the readiness check from
`es.ping()` (fails silently) to `es.cluster.health()` (raises an
explicit error), for clearer diagnostics if it happens again.

Lesson for future debugging: if an external health check (curl) passes
but the application client still fails to connect, check dependency
version compatibility before adjusting timing or retry logic.
